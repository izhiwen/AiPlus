# v0.3.0 Polish Patch Implementation Notes

## Phase 1 Design

### Item 1: coordinator_decision dispatch-log event

Every adaptive route score writes one JSONL event to `.aiplus/agents/dispatch-log.jsonl`, including `LIGHT_NO_CODE` and `--score-only` runs that do not dispatch staff.

Schema:

```json
{
  "schemaVersion": "0.3.0",
  "event": "coordinator_decision",
  "decisionId": "coord-<epoch-ms>",
  "timestamp": "<ISO-8601>",
  "mode": "route|score_only",
  "taskExcerpt": "<redacted, max 200 chars>",
  "complexity": 1,
  "risk": 0.15,
  "tier": "LIGHT_NO_CODE",
  "codeChange": false,
  "designImpact": false,
  "consultant": "skip|fire",
  "staffingRoles": [],
  "dispatched": false,
  "secretValues": "none"
}
```

`taskExcerpt` is whitespace-normalized, capped at 200 characters plus `...`, and redacts common API-token/password/key shapes before write. Existing per-role dispatch rows are unchanged.

### Item 2: doctor BWS runtime-auth warning

`aiplus agent doctor` emits `WARN_SECRET_BROKER_RUNTIME_AUTH` when all conditions are true:

- `aiplus secret-broker status` equivalent state reports `provider=bws`
- the active agent team has at least one active role
- neither `ANTHROPIC_API_KEY` nor `OPENAI_API_KEY` is present in the current process environment

Warning copy:

```text
  WARN_SECRET_BROKER_RUNTIME_AUTH
  Detected BWS backend + active agent-team roles, but no provider key in env.
  Adapter dispatch will fail with auth error.
  Fix: wrap your dispatch with
    aiplus secret-broker run --aliases anthropic,openai -- aiplus agent route "<task>"
  Or set ANTHROPIC_API_KEY / OPENAI_API_KEY in your shell once.
```

This is WARN only. It does not fail doctor because keyring-backed setups, local-only roles, and Builder environments may remain valid.

### Item 3: scorer calibration fixture

Fixture: `crates/aiplus-cli/tests/fixtures/coordinator_calibration.toml`.

Coverage matrix:

- `LIGHT_NO_CODE`: status/explain/summarize prompts, including T3 baseline
- `LIGHT_CODE`: small patch/update/delete prompts
- `MEDIUM`: implementation, workflow, API/integration, architecture/design prompts
- `HEAVY`: auth/security/payment/deploy/release prompts
- boundary coverage: complexity 2 to 3, risk 0.65 below heavy threshold, risk 0.75 above heavy threshold

The fixture asserts exact `complexity` and `tier`, and `risk <= expected_risk_max`. This locks current behavior without re-deriving the rubric.

### Item 4: route --score-only

`aiplus agent route --score-only "<task>"` and Chinese alias `--打分`:

- runs coordinator scoring and dry-run staffing only
- prints the adaptive coordinator line
- prints `Plan step: ...` with consultant dry-run behavior
- prints `Would staff: [...]`
- does not call consultant, dispatch agents, provision worktrees, or write disk cache
- writes a `coordinator_decision` dispatch-log row with `mode=score_only`, `dispatched=false`
- exits 0

For normal adaptive route runs, the same event writes with `mode=route`; `dispatched=false` for `LIGHT_NO_CODE`, `true` for staffed tiers.
