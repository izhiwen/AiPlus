## 1. TTL Enforcement Algorithm

Disk cache remains opt-in through `[cache] enable_disk = true`. v0.3.1 adds
`[cache] enforce_ttl = false` with a conservative default of `false` so
existing installations keep current warm-cache behavior unless the Owner opts
in.

TTL age is calculated from the cache metadata `last_used_at_ms` field, not the
snapshot write time, because the configured TTL is meant to describe bench
freshness since last use.

Pseudocode:

```text
if enable_disk == false:
  return Disabled, ttl_expired = null

meta = load project cache meta
entry = meta.roles[role]
if no entry or no snapshot:
  return ColdStart, ttl_expired = false

age_ms = now_ms - entry.last_used_at_ms
expired = age_ms > role.warm_bench_ttl_seconds * 1000

if enforce_ttl == true and expired:
  meta.invalidate(role, TtlExpired)
  return ColdStart, ttl_expired = true

return DiskWarm, ttl_expired = expired
```

`coordinator_decision.ttl_expired` is `null` for `--score-only`, `false` when
there are no staffed roles or no expired staffed cache entry, and `true` when
any staffed role has an expired cache entry. Direct `agent route <role>` keeps
the existing cache-source output and metrics path.

`aiplus doctor` reports `[cache] enforce_ttl` status. When cache metadata
exists, it prints per-role `INFO` lines with cache age and TTL; with
`enforce_ttl=true`, expired entries also produce `WARN` lines.

## 2. Risk-Forced Staffing Rule

Risk forcing is a deterministic post-processing step after the existing
baseline tier staffing and before expert auto-summoning.

| Tier | Risk band | Forced delta |
| --- | --- | --- |
| LIGHT_NO_CODE / LIGHT_CODE | risk < 0.70 | none |
| LIGHT_NO_CODE / LIGHT_CODE | 0.70 <= risk < 0.85 | ensure `reviewer` |
| LIGHT_NO_CODE / LIGHT_CODE | risk >= 0.85 | ensure `reviewer`, ensure `qa` |
| MEDIUM / HEAVY | any risk | same thresholds, but only record roles that were not already present |

`forced_by_risk` records the roles required by the risk threshold, while the
staffing vector is deduped. This makes risk intent visible in
`coordinator_decision` even when HEAVY baseline staffing already includes
`reviewer` and `qa`.

## 3. Auto-Summon Registry And Keyword Match

The registry is built from loaded team role TOML files. A missing
`[autosummon]` section is valid and means the role is never auto-summoned.
This preserves backward compatibility for installed teams and third-party
roles.

Schema:

```toml
[autosummon]
keywords = ["security", "secret", "auth"]
match_mode = "any"
priority = 90
```

Algorithm:

```text
registry = active roles where autosummon.keywords is non-empty
task_text = lowercase task

for role in registry:
  hits = keywords whose lowercase form appears in task_text
  if match_mode == "all": matched = hits.len == keywords.len
  else: matched = hits.len > 0
  if matched and role not already staffed:
    candidate(role, priority)

sort candidates by priority desc, then role asc
append candidates until tier cluster cap is reached
```

## 4. Initial Expert Trigger Set

Initial v0.1 expert coverage uses three active AiPlus experts:

- `security-reviewer`: `security`, `secure`, `auth`, `secret`, `token`,
  `credential`, `payment`, `billing`, `privacy`, `vulnerability`, `csrf`,
  `xss`, `encryption`.
- `tech-writer`: `docs`, `documentation`, `readme`, `guide`, `manual`,
  `onboarding`, `copy`, `release notes`, `spec`, `tutorial`.
- `ai-integration`: `llm`, `ai`, `model`, `prompt`, `rag`, `embedding`,
  `openai`, `anthropic`, `claude`, `codex`, `agent`, `token budget`.

All three use `match_mode = "any"`. Priorities are security-reviewer 90,
ai-integration 70, and tech-writer 60 so security-sensitive tasks win scarce
LIGHT capacity before documentation-only experts.

## 6. Schema Migration

`AgentConfig` gains optional `autosummon: Option<AutosummonSection>`.
Deserialization uses `#[serde(default)]`, so absent `[autosummon]` sections
skip registry inclusion rather than failing config load. Existing installed
TOML files remain valid.

The template TOMLs for `security-reviewer`, `tech-writer`, and
`ai-integration` receive the new section so future installs include the
registry data.

## 7. Cluster-Cap Rule

After baseline staffing, risk forcing, and auto-summon candidate ordering, the
final staffing vector is capped by tier:

- LIGHT_NO_CODE: max 2 roles.
- LIGHT_CODE: max 3 roles.
- MEDIUM: max 5 roles.
- HEAVY: max 8 roles.

Baseline and risk-forced roles are preserved first. Auto-summoned experts fill
remaining capacity in priority order. Duplicate roles are ignored.
