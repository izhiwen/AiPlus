Status: in progress

## 1. Scoring Rubric

P0 implements scoring inside `crates/aiplus-cli/src/agent/coordinator.rs`.
The source of truth is `aiplus-agent-team/DESIGN.md` section 9, with the
audit memo's decision not to reuse `aiplus-auto-team-consultant`.

Complexity uses the DESIGN scale 1-5:

- 1: status, explanation, read-only question, or single known command.
- 2: small localized code/doc change with one owner and low ambiguity.
- 3: multi-step implementation, test update, or route through one engineer
  plus reviewer.
- 4: cross-module feature, migration, integration, or design-impacting work.
- 5: payment/auth/security/release/data-loss/high-blast-radius work, or work
  that naturally needs PM, Architect, two Engineers, Reviewer, and QA.

Risk uses the DESIGN scale 0.0-1.0:

- 0.10-0.29: read-only/status/docs without external side effects.
- 0.30-0.49: ordinary local code/test change.
- 0.50-0.69: cross-module, migration, CI/release, data, or dependency risk.
- 0.70-0.90: payment, auth, security, privacy, secrets, production, deploy,
  data loss, rollback, compliance, or Owner-gated risk.

The D5 payment task `"实现支付接口"` must score complexity 4-5 and risk
0.7-0.9. P0 pins it to complexity 5 and risk 0.85 so the HEAVY path is
deterministic.

## 2. Tier Classifier

Tier classification follows DESIGN.md section 9.2:

- LIGHT no-code: complexity <= 2 and no file write expected.
- LIGHT code: complexity <= 2 and a file write is expected.
- MEDIUM: complexity 3-4.
- HEAVY: complexity >= 5 or risk >= 0.70.

HEAVY wins over all lower tiers. The risk boundary is inclusive: risk 0.70
is HEAVY, risk 0.69 is not HEAVY unless complexity is 5.

## 3. Staffing Dispatcher

P0 staffing is deterministic and limited to core roles:

- LIGHT no-code: CEO/PI handles directly; no worktree staffing.
- LIGHT code: `engineer-a`.
- MEDIUM: `engineer-a`, `reviewer`; add `architect` when the scorer marks
  design impact.
- HEAVY: `pm`, `architect`, `engineer-a`, `engineer-b`, `reviewer`, `qa`.

P1 expert auto-summoning is explicitly deferred. P0 does not add
`security-reviewer` automatically even when risk >= 0.70; that is called out
in the briefing as out of scope for v0.3.0 P0.

## 4. Two-Step Flow

LIGHT tasks skip consultant for speed.

MEDIUM and HEAVY tasks follow the two-step CEO flow:

1. Plan step: run the existing consultant side effect with `run_consult`.
   This reads `.aiplus/consultant-team.toml` and writes read-only findings.
2. Execute step: dispatch the selected staffing roles through the existing
   `agent route` role path so worktree provisioning, audit logging, owner
   gates, cache invalidation, and dispatch metrics remain centralized.

## 5. Deviation Note

DESIGN.md section 9.1 says the CEO reuses Auto Team Consultant's scoring.
The 2026-05-18 audit memo rejects direct reuse for v0.3.0 because the
consultant module has no stable Rust scoring API, uses inconsistent scoring
scales, and is pre-1.0. P0 therefore implements scoring internally in
`crates/aiplus-cli/src/agent/coordinator.rs` and treats consultant as a
downstream read-only advice step, not the upstream scoring source.
