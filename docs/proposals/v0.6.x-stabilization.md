## TL;DR

v0.6.0 shipped major user-visible features: G1 natural-language role
triggers, G2 semantic dispatch gate, Perf-1 dispatch acceleration.
Before pushing more new functionality (G3 / new modules / new runtime
adapters), dedicate one milestone to closing known gaps and polishing
the existing surface. This raises the floor for adoption and reduces
the time new users spend hitting edge cases.

## Goals (D1-D6, Owner-signed when approved)

- **D1**: Close all still-relevant P0 gaps from
  `docs/roadmap/v0.5.2-known-gaps.md`.
- **D2**: Catch up on Windows test parity — current matrix is
  macOS + Linux focused (known-gaps #6).
- **D3**: Localize key v0.6.0+ docs into Chinese (parity for the
  major user-facing surface).
- **D4**: Promote any remaining DRAFT proposal docs to APPROVED if
  their implementation has shipped.
- **D5**: Clean up the 37 clippy warnings flagged by the recent
  reviewer A pass (all auto-fixable style/perf categories).
- **D6**: One round of local branch cleanup in `aiplus-public` (a
  number of merged feature branches still linger).

## Non-goals

- No new product features (no G3 — that is a separate spec / separate
  milestone after this one).
- No new runtime adapters; Codex / Claude Code / OpenCode is enough.
- No breaking changes to the public CLI surface.
- Not promising specific delivery dates — this is
  opportunistic-priority polish work, not a roadmap commitment.

## Scope (prioritized)

### P0 — must do this milestone

1. **Capsule recovery edge cases** (`v0.5.2-known-gaps.md` #1).
   `compact resume` currently fails on malformed UTF-8 inside the
   capsule. Fix or wrap in a hardened recovery path that surfaces a
   clear error.
2. **Velocity retention policy** (`v0.5.2-known-gaps.md` #2). No
   automatic purge of velocity records > 90 days; long-running
   projects accrue unbounded disk. Add a configurable purge.
3. **Cross-module conflict detection in `aiplus doctor`**
   (`v0.5.2-known-gaps.md` #3). Doctor scans modules in isolation;
   schema conflicts between compact-reminder and agent-memory go
   silent. Add cross-check.
4. **Human-readable doctor output**. `aiplus doctor` currently
   produces machine-friendly `key=VALUE` lines that intimidate
   non-technical users (e.g. `NEEDS_FIX nl_role_triggers=FAIL_AGENTS_CATALOG_STALE`).
   Add a `--explain` mode (default ON when stdout is a TTY, OFF in
   pipes) that pairs each `NEEDS_FIX` / `FAIL` line with a
   human-language explanation and a concrete `next-action` hint.
   Backward-compatible: scripts grepping for `NEEDS_FIX` / `PASS`
   keywords still work. Build a catalog for the top 10 failure
   modes; bilingual EN + ZH where possible. Examples:

   - `FAIL_AGENTS_CATALOG_STALE` →
     *"Your project's role catalog is out of date.*
     *Fix: run `aiplus install <runtime>` to refresh."*
   - `FAIL_MISSING_AGENTS_CATALOG` →
     *"AiPlus isn't installed in this directory yet.*
     *Fix: run `aiplus install <runtime>`."*
   - `NEEDS_FIX manifest schemaVersion supported` →
     *"AiPlus's saved manifest needs migrating.*
     *Fix: run `aiplus doctor --fix` to update automatically."*

   This is the single highest-impact UX change for new-user
   onboarding (per Advisor's non-programmer readability audit
   2026-05-17). Move user-facing copy into a single message catalog
   (`crates/aiplus-cli/src/messages.rs`) so it can be translated
   and reviewed independently of code.

### P1 — should do

4. **Redaction multi-language coverage** (`v0.5.2-known-gaps.md` #4).
   The `raw transcript` pattern only catches English; non-English
   transcripts may leak into memory records.
5. **Consultant team TOML validation at runtime**
   (`v0.5.2-known-gaps.md` #5). Invalid consultant configs are
   silently ignored. Enforce schema at load.
6. **`is_supported_manifest_schema` CI test for the future-major
   branch**. v0.6.0 release prep hit a "future major rejected" issue
   because `0.6.0` wasn't yet in the supported set; add a CI smoke
   test that fails fast when crate version is bumped without a
   matching schema prefix update.
7. **`aiplus quickstart` — guided setup command**. New top-level
   subcommand that walks first-time users through install +
   `secret-broker` + first-run via interactive TTY prompts. Three
   questions: which AI coding agent you use, AEL or not, install
   destination. Then sequences existing commands and prints a
   "try this next" hint. Falls back to `aiplus install all` with
   sensible defaults in non-TTY mode. Eliminates the need for new
   users to read CLI documentation for the setup phase.

   Implementation: use `dialoguer` or `inquire` crate for prompts;
   wraps existing `install` / `add` / `secret-broker` subcommands.
   Output example:

   ```
   $ aiplus quickstart

   Welcome to AiPlus! Let's get you set up in 3 questions.

   [1/3] Which AI coding agent are you using?
     > Claude Code
       Codex
       OpenCode
       I don't have one yet (show install links)

   [2/3] Do you want AEL (applied-economics research team)?
     > Yes (I'm a researcher / academic)
       No (just software engineering)

   [3/3] Where should AiPlus install in this project?
     > /current/directory (auto-detected)

   Setting up...
   ✓ Installed claude-code adapter
   ✓ Set up secret-broker (your key stays in OS keychain)
   ✓ Added AEL research-team module
   ✓ Verified everything with aiplus doctor

   All set! Try: `ael talk advisor "What is your role?"`
   ```

   Acceptance: a new user with an AI coding agent already
   installed can run `aiplus quickstart` and have a working setup
   in under 2 minutes. (Source: Owner's non-programmer
   accessibility audit 2026-05-17, three quick wins.)

### P2 — nice to have

## Phases

- **Phase A** (P0): capsule recovery + velocity purge +
  cross-module doctor. Estimated ~2-3 weeks.
- **Phase B** (P1): redaction multi-language + consultant TOML
  validation + manifest-schema CI test. Estimated ~1-2 weeks.
- **Phase C** (P2): Windows tests + localization + clippy cleanup.
  Opportunistic, spread out, estimated ~1-2 weeks total.

## Acceptance criteria

- All P0 items have shipped fixes + tests.
- `aiplus doctor` passes cleanly on macOS + Linux + Windows after
  Phase A + C.
- Translated docs available for the v0.6.0 release notes and
  README at minimum.
- Clippy warnings count drops to < 5.
- No new test failures introduced.
- CHANGELOG entry in v0.6.x release(s) documents each closure.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Some "known gaps" are already addressed in v0.5.27 / v0.5.28 (which weren't tracked in v0.5.2-known-gaps.md) | First task of this milestone: re-audit the known-gaps doc against current main |
| Windows CI setup is non-trivial (Mac dev with no Windows runner experience) | Phase C; can defer if blocked, doesn't gate Phase A/B |
| `cargo clippy --fix` introduces subtle behavior changes | Manual review of every auto-fix diff + full test suite after |
| Doctor cross-module check has surprise interactions with installed projects | Implement gated behind a feature flag first; promote to default after one release cycle |

## What's NOT in this milestone

- **G3** (next major gate / feature) — separate spec, separate
  milestone after this stabilization.
- **OpenCode TUI automation (G1.1 V1.1)** — still blocked on upstream
  OpenCode ready-marker stabilization.
- New runtime adapters.
- Major refactor or architecture changes.
- Community / outreach work — separate stream, tracked elsewhere.

## When to declare done

When acceptance criteria are met AND the next consecutive Advisor /
CEO retrospective reports zero P0 from this milestone as still open.
