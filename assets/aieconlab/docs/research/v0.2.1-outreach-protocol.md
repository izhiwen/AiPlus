# AEL v0.2.1 Outreach Protocol

This protocol is a scaffold for the v0.2.1 user-research pilot. It does not
authorize automated recruiting, email sending, interview scheduling, or hosted
telemetry collection. Those are Owner actions.

## Pilot Goals

- Learn whether applied economists understand AEL as a standing research team,
  not a generic chatbot wrapper.
- Identify which roles are useful in real research work and which role names
  confuse users.
- Test whether explicit telemetry opt-in is clear enough for users working
  with unpublished papers and private data.
- Collect early friction in install, first route, and first role-specific
  conversation.

## Cohorts

- 5 interview participants: applied economists, economics PhD students, or RAs
  who already use AI tools for coding, writing, or literature work.
- 20 telemetry testers: users willing to try AEL on a non-sensitive project and
  opt in to local telemetry review.

Recruiting should prioritize users with an active research project, but the
first interview should use a toy or public-data project to validate the script.

## Interview Script

1. What kind of economics research work do you use AI tools for today?
2. When you hear "permanent virtual research team," what do you expect AEL to
   do?
3. Which role would you ask first for your current project: Advisor, PI,
   Theorist, RA-Stata, Referee, Replicator, or another expert?
4. Run the install and first `ael talk advisor "What is your role?"` flow on a
   disposable project. Where did you hesitate?
5. Route one real but non-sensitive task. Which role did AEL choose, and was it
   the role you expected?
6. Would local opt-in telemetry be acceptable if it records only role dispatch,
   task class, and success or failure?
7. What information would make you decline telemetry, even if it stays local?
8. What would make AEL credible enough to try on a live paper?

## Recruiting Templates

Short email:

```text
Subject: 20-minute feedback request on AiEconLab

I am testing AiEconLab, an open-source AI research-team layer for applied
economists. I am looking for early feedback from economists who already use AI
tools in research workflows.

The session is 20 minutes, uses a toy or non-sensitive project, and does not
require sharing private data. I will ask you to try the install and one role
conversation, then react to the workflow and privacy model.

If you are open to it, please send two times that work this week.
```

Telemetry tester note:

```text
I am looking for early AEL testers willing to opt in to local-only telemetry.
For v0.2.1, AEL writes a local JSON config and local event file only. No hosted
endpoint is configured. The intended event shape is role dispatched, task class,
success or failure, and timestamp. Do not use private data for the pilot.
```

## Privacy Notice

AEL telemetry is off by default. Users must run `ael telemetry enable` before
any telemetry pilot should be treated as active. For v0.2.1, telemetry is a
local scaffold:

- Config path: `.ael/telemetry.json`
- Event path: `.ael/telemetry-events.jsonl`
- Hosted endpoint: none
- Payload fields: expert, task_class, outcome, timestamp
- Excluded data: prompts, paper text, code, data paths, filenames, model
  output, personal identifiers, and repository remotes

Users can run `ael telemetry disable` at any time. Disabling writes the local
config with `enabled: false`; it does not upload, delete, or transmit anything.

## Event Payload Spec

```json
{
  "expert": "referee",
  "task_class": "HEAVY",
  "outcome": "success",
  "timestamp": "2026-05-17T00:00:00Z"
}
```

Valid `task_class` values are `LIGHT`, `MEDIUM`, `HEAVY`, and `unknown`.
Valid `outcome` values are `success`, `failure`, and `unknown`.

## Owner Handoff

scaffold. Do not ask testers to use private data or unpublished drafts in the
pilot. Hosted telemetry collection is Tier 2.2 work and is not part of
v0.2.1.
